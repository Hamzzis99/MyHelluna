## 1. 아키텍처/설계 문제

### 🔴 1-1. God-Class: Inv_InventoryComponent.cpp (~2600줄)
- **위치**: `InventoryManagement/Components/Inv_InventoryComponent.cpp`
- **문제**: 20개 이상의 Server RPC, 저장/로드, 장비 장착/해제, 크래프팅, 컨테이너, 부착물, 드롭, 스택 관리 등 모든 인벤토리 로직이 단일 클래스에 집중
- **영향**: 유지보수 어려움, 컴파일 시간 증가, 테스트 불가능
- **권장**: 기능별 서브컴포넌트로 분리 (CraftingSubsystem, EquipSubsystem, ContainerSubsystem 등)

### 🔴 1-2. God-Class: Inv_InventoryGrid.cpp (~4500줄)
- **위치**: `Widgets/Inventory/Spatial/Inv_InventoryGrid.cpp`
- **문제**: Grid 렌더링, 드래그 앤 드롭, 스택 관리, Split, 회전, 부착물 패널, 저장/복원, 로비 전송, 컨테이너 전송, 팝업 메뉴 등 모든 Grid UI 로직이 단일 클래스에 집중
- **영향**: 약 80,000 토큰 (단일 파일 기준 프로젝트 최대), 변경 시 사이드이펙트 추적 곤란
- **권장**: InputHandler, StateManager, TransferManager 등으로 분리

### 🟡 1-3. God-Class: Inv_PlayerController.cpp (~1900줄)
- **위치**: `Player/Inv_PlayerController.cpp`
- **문제**: Tick (LineTrace), 인벤토리 저장/로드/복원, 청크 전송, 컨테이너 UI, HUD 등 담당
- **영향**: PlayerController의 비대화로 인한 결합도 증가
- **권장**: 인터랙션, 저장/로드를 별도 컴포넌트로 분리

## 2. 매 프레임 실행 (Tick/NativeTick) 문제

### 🔴 2-1. PlayerController::Tick — 매 프레임 LineTrace
- **위치**: `Inv_PlayerController.cpp` Tick()
- **문제**: `PrimaryActorTick.bCanEverTick = true`로 항상 활성. `TraceForInteractables()`가 매 프레임 LineTrace 실행
- **영향**: 인벤토리 메뉴가 열려 있어도 Tick 자체는 계속 실행됨 (early return하지만 함수 호출 오버헤드)
- **권장**: 타이머 기반 간격 트레이스 (0.1~0.2초), 또는 Overlap 기반 감지로 전환

### 🟡 2-2. BuildingComponent::TickComponent — 매 프레임 실행
- **위치**: `Building/Inv_BuildingComponent.cpp`
- **문제**: `TickComponent`가 매 프레임 실행. 내부에서 `CloseCraftingMenuIfOpen()` 호출 시 `GetAllWidgetsOfClass()` + 문자열 `Contains()` 검색 수행
- **영향**: `GetAllWidgetsOfClass()`는 UWidgetTree 전체를 순회하는 O(n) 연산
- **권장**: 위젯 참조를 캐싱, Tick 대신 이벤트/델리게이트 사용

### 🟡 2-3. SpatialInventory::NativeTick — ItemDescription 위치 갱신
- **위치**: `Widgets/Inventory/Spatial/Inv_SpatialInventory.cpp`
- **문제**: 인벤토리가 열려 있는 동안 매 프레임 `ItemDescription` 위젯 위치를 마우스 기준으로 갱신
- **권장**: 마우스 이동 이벤트 기반으로 전환, 또는 Slate의 SetPosition을 직접 사용

### 🟡 2-4. AttachmentPanel::NativeTick — 슬롯 하이라이트
- **위치**: `Widgets/Inventory/AttachmentSlots/Inv_AttachmentPanel.cpp`
- **문제**: 패널이 열려 있는 동안 매 프레임 `UpdateSlotHighlights()` 실행
- **권장**: 마우스 Hover/Leave 이벤트 기반으로 전환

### 🟢 2-5. CraftingButton / BuildingButton — 빈 NativeTick
- **위치**: `Widgets/Crafting/Inv_CraftingButton.cpp`, `Widgets/Building/Inv_BuildingButton.cpp`
- **문제**: `NativeTick()` 함수가 존재하지만 내부 로직 없음 (Super::NativeTick만 호출)
- **영향**: Tick 등록 오버헤드 (미미하지만 불필요)
- **권장**: NativeTick 오버라이드 삭제

### 🟢 2-6. CharacterDisplay::NativeTick — 드래그 회전
- **위치**: `Widgets/CharacterDisplay/Inv_CharacterDisplay.cpp`
- **문제**: 매 프레임 실행되지만 `!bIsDragging` 시 early return
- **영향**: 낮음 (빠른 early return)
- **권장**: 드래그 시작/종료 시 Tick 활성화/비활성화

## 3. 네트워크/리플리케이션 문제

### 🔴 3-1. DOREPLIFETIME 조건 없음 — EquipActor 6개 프로퍼티
- **위치**: `EquipmentManagement/Inv_EquipActor.cpp`
- **문제**: 6개 리플리케이션 프로퍼티(`MeshPath`, `AttachSocketName`, `bSilencer`, `bScope`, `bLaser`, `AttachmentVisuals`)가 `DOREPLIFETIME`으로 매 리플리케이션 번들에 포함
- **영향**: 무기 상태가 변경되지 않아도 매번 전체 프로퍼티 전송 → 대역폭 낭비
- **권장**: `DOREPLIFETIME_CONDITION(COND_None → COND_InitialOrOwner / COND_OwnerOnly)`

### 🟡 3-2. OnRep_AttachmentVisuals — 전체 파괴/재생성
- **위치**: `EquipmentManagement/Inv_EquipActor.cpp`
- **문제**: `OnRep_AttachmentVisuals()`가 호출될 때마다 기존 부착 메시 전체 파괴 후 재생성
- **영향**: 슬롯 하나만 변경해도 모든 부착물 재생성 → GC 부담 + 프레임 스파이크
- **권장**: Diff 기반 업데이트 (변경된 슬롯만 갱신)

### 🟡 3-3. Reliable Multicast 사운드 — ResourceComponent
- **위치**: `Resource/Inv_ResourceComponent.cpp`
- **문제**: `Multicast_PlaySoundAtLocation()`이 `Reliable`로 마킹됨
- **영향**: 사운드 재생은 손실 허용 가능한데 Reliable Queue를 사용하여 네트워크 버퍼 점유
- **권장**: `Unreliable`로 변경

### 🟡 3-4. bAlwaysRelevant=true — BuildingActor
- **위치**: `Building/Inv_BuildingActor.cpp`
- **문제**: 건물 액터가 `bAlwaysRelevant = true` 설정
- **영향**: 맵 반대편의 건물도 항상 리플리케이트 → 건물 수 증가 시 대역폭 선형 증가
- **권장**: 건물은 거리 기반 Relevancy가 적합, `NetCullDistanceSquared` 사용

### 🟢 3-5. bCaptureEveryFrame=true — WeaponPreviewActor
- **위치**: `Interaction/Inv_WeaponPreviewActor.cpp`
- **문제**: SceneCaptureComponent2D의 `bCaptureEveryFrame = true`로 패널이 열려있는 동안 매 프레임 캡처
- **영향**: GPU 비용 (전체 씬 렌더링 복사)
- **권장**: 무기/부착물 변경 시에만 `CaptureScene()` 수동 호출

## 4. 캐시/조회 비효율

### 🔴 4-1. RebuildItemTypeIndex() — O(n) 전체 재구축
- **위치**: `InventoryManagement/FastArray/Inv_FastArray.cpp`
- **문제**: 아이템 추가/제거 시마다 `ItemTypeIndex` (TMultiMap) 전체를 클리어 후 재구축
- **영향**: 인벤토리 아이템 수에 비례하는 O(n) 비용이 매 변경마다 발생
- **권장**: 증분 업데이트 — `AddEntry` 시 해당 태그만 Add, `RemoveEntry` 시 해당 태그만 Remove

### 🟡 4-2. GetItemManifest() 값 복사 반환 — ItemComponent
- **위치**: `Items/Components/Inv_ItemComponent.h`
- **문제**: `GetItemManifest()`와 `GetItemManifestMutable()`가 `FInv_ItemManifest`를 **값으로 반환** (by value)
- **영향**: `FInv_ItemManifest`는 `TArray<TInstancedStruct<FInv_ItemFragment>>` + `TMap<UScriptStruct*, int32>` 캐시를 포함하는 중량 구조체. 호출마다 전체 복사
- **권장**: `const FInv_ItemManifest&` 참조 반환

### 🟡 4-3. FindComponentWithSocket — 선형 검색
- **위치**: `EquipmentManagement/Inv_EquipActor.cpp`
- **문제**: `FindComponentWithSocket()`이 액터의 모든 SceneComponent를 순회하며 소켓 이름 매칭
- **영향**: 컴포넌트 수에 비례하는 O(n)
- **권장**: 소켓→컴포넌트 매핑을 BeginPlay에서 캐싱

### 🟡 4-4. CraftingButton/BuildingButton — GetAllItems() 3회 호출
- **위치**: `Widgets/Crafting/Inv_CraftingButton.cpp`, `Widgets/Building/Inv_BuildingButton.cpp`
- **문제**: `UpdateMaterialUI()`에서 재료 1/2/3 각각에 대해 `GetAllItems()`를 별도 호출 (총 3회)하고 각각 전체 순회
- **영향**: 동일한 아이템 목록을 3번 복사 + 3번 순회 = O(3n)
- **권장**: 한 번의 `GetAllItems()` 호출로 3개 재료를 동시 검사

### 🟡 4-5. GetAllActorsOfClass — CharacterDisplay 초기화
- **위치**: `Widgets/CharacterDisplay/Inv_CharacterDisplay.cpp`
- **문제**: `NativeOnInitialized()`에서 `GetAllActorsOfClass()` 호출
- **영향**: 월드의 모든 액터를 순회하는 O(n) 연산
- **권장**: 직접 참조 설정 또는 태그 기반 검색

### 🟡 4-6. HellunaItemTypeMapping — DataTable 전체 순회
- **위치**: `Source/Helluna/Private/Inventory/HellunaItemTypeMapping.cpp`
- **문제**: `GetActorClassFromItemType()`이 `GetAllRows()`로 전체 Row를 가져온 후 순차 검색
- **영향**: DataTable Row 수에 비례하는 O(n)
- **권장**: `FindRow<>()` 사용, 또는 TMap 캐시 구축

### 🟢 4-7. IsSlotOccupied — 선형 검색 (AttachmentFragments)
- **위치**: `Items/Fragments/Inv_AttachmentFragments.cpp`
- **문제**: `IsSlotOccupied()`가 `AttachedItems` 배열을 선형 검색 (SlotIndex 매칭)
- **영향**: 부착물 슬롯 수가 적어 현재 영향 미미 (보통 2~4개)
- **권장**: 향후 슬롯 수 증가 시 TMap<int32, FInv_AttachedItemData> 전환 고려

## 9. 긍정적 최적화 (이미 적용된 것)

확인된 좋은 패턴:
- **OccupiedMask 비트배열** (InventoryGrid): `IsAreaFree()` O(1) 빠른 공간 체크
- **SlottedItem 풀링** (InventoryGrid): `AcquireSlottedItem()`/`ReleaseSlottedItem()` 위젯 재활용
- **FragmentTypeCache** (ItemManifest): `TMap<UScriptStruct*, int32>`로 O(1) Fragment 조회
- **ItemTypeIndex** (FastArray): `TMultiMap<FGameplayTag, int32>`로 O(1) 타입별 조회 (증분 업데이트만 부족)
- **bShouldTickForHover** (InventoryGrid): HoverItem 없으면 Tick early return
- **Chunk-based RPC** (PlayerController): 65KB RPC 제한 우회, 누적 버퍼 패턴
- **WeakObjectPtr 사용**: 대부분의 컴포넌트 참조에 TWeakObjectPtr 적용
- **서버 Validate 함수**: 모든 Server RPC에 `_Validate` 함수 구현
- **NativeDestruct 바인딩 해제**: 대부분의 위젯에서 델리게이트 정리 구현
