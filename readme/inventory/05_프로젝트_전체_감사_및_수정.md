## 1. HellunaFunctionLibrary ASC null 체크

### 파일
`Source/Helluna/Private/HellunaFunctionLibrary.cpp`

### 수정 전
```cpp
void UHellunaFunctionLibrary::AddGameplayTagToActorIfNone(AActor* InActor, FGameplayTag TagToAdd)
{
    UHellunaAbilitySystemComponent* ASC = NativeGetHellunaASCFromActor(InActor);
    if (!ASC->HasMatchingGameplayTag(TagToAdd))  // ASC가 nullptr이면 크래시
    {
        ASC->AddLooseGameplayTag(TagToAdd);
    }
}
// RemoveGameplayTagFromActorIfFound, NativeDoesActorHaveTag도 동일 패턴
```

### 수정 후
```cpp
void UHellunaFunctionLibrary::AddGameplayTagToActorIfNone(AActor* InActor, FGameplayTag TagToAdd)
{
    UHellunaAbilitySystemComponent* ASC = NativeGetHellunaASCFromActor(InActor);
    if (!ASC) return;  // null 안전

    if (!ASC->HasMatchingGameplayTag(TagToAdd))
    {
        ASC->AddLooseGameplayTag(TagToAdd);
    }
}
// RemoveGameplayTagFromActorIfFound → if (!ASC) return;
// NativeDoesActorHaveTag → if (!ASC) return false;
```

### 문제였던 이유
- `NativeGetHellunaASCFromActor()`는 ASC가 없는 액터에서 nullptr을 반환
- 반환값 검증 없이 `ASC->HasMatchingGameplayTag()` 호출 → nullptr 역참조 크래시
- 3개 함수 모두 동일한 패턴으로 위험했음

### 테스트 방법
```
환경: 에디터 PIE
수정 전 비교:
  1. ASC가 없는 액터(StaticMeshActor 등)에 Blueprint에서 AddGameplayTagToActorIfNone 호출
  2. 수정 전: 즉시 크래시 (Access violation)
  3. 수정 후: 함수가 조용히 return, 크래시 없음

정상 동작 확인:
  1. 히어로 캐릭터(ASC 있음)에 GameplayTag 추가/제거
  2. 기존과 동일하게 정상 동작해야 함
```

## 2. RepairWidget 델리게이트 해제

### 파일
`Source/Helluna/Private/Widgets/RepairWidget.cpp`

### 수정 전
```cpp
void URepairWidget::NativeDestruct()
{
    Super::NativeDestruct();
    // 바인딩한 Button/Slider 델리게이트 해제 없음 → 메모리 누수
}
```

### 수정 후
```cpp
void URepairWidget::NativeDestruct()
{
    if (Button_Confirm)
        Button_Confirm->OnClicked.RemoveDynamic(this, &URepairWidget::OnConfirmClicked);
    if (Button_Cancel)
        Button_Cancel->OnClicked.RemoveDynamic(this, &URepairWidget::OnCancelClicked);
    if (Slider_Material1)
        Slider_Material1->OnValueChanged.RemoveDynamic(this, &URepairWidget::OnMaterial1SliderChanged);
    if (Slider_Material2)
        Slider_Material2->OnValueChanged.RemoveDynamic(this, &URepairWidget::OnMaterial2SliderChanged);

    Super::NativeDestruct();
}
```

### 문제였던 이유
- NativeConstruct에서 `AddDynamic`으로 4개 델리게이트 바인딩
- NativeDestruct에서 `RemoveDynamic` 없이 Super만 호출
- 위젯이 파괴되어도 델리게이트 참조가 남아 GC가 수거 못 함 → 메모리 누수
- 반복 열기/닫기 시 누적

### 테스트 방법
```
환경: 에디터 PIE (As Client)
수정 전 비교:
  1. F키로 RepairWidget 열기 → 닫기를 20회 반복
  2. 수정 전: stat UObject에서 RepairWidget 관련 오브젝트 수가 계속 증가
  3. 수정 후: 위젯 수가 안정적 (열고 닫아도 누적되지 않음)

정상 동작 확인:
  1. F키로 열기 → 슬라이더 조작 → 확인/취소 버튼 클릭
  2. 버튼/슬라이더가 정상 반응해야 함
  3. 닫은 후 다시 열어도 정상 동작
```

## 3. Equipmnt → Equipment 네임스페이스 오타 수정

### 파일
`Plugins/Inventory/Source/Inventory/Public/Items/Inv_ItemTags.h`

### 수정 전
```cpp
// Inv_ItemTags.h (헤더)
namespace GameItems
{
    namespace Equipmnt  // ← 오타!
    {
        namespace Weapons { ... }
        namespace Cloaks { ... }
        namespace Masks { ... }
    }
}

// Inv_ItemTags.cpp (구현)
namespace GameItems
{
    namespace Equipment  // ← 정상
    {
        namespace Weapons { ... }
    }
}
```

### 수정 후
```cpp
// Inv_ItemTags.h (헤더) — .cpp와 일치
namespace GameItems
{
    namespace Equipment  // ← 수정됨
    {
        namespace Weapons { ... }
        namespace Cloaks { ... }
        namespace Masks { ... }
    }
}
```

### 문제였던 이유
- .h에서 `Equipmnt`로 선언, .cpp에서 `Equipment`로 정의 → 네임스페이스 불일치
- 현재 코드에서 `GameItems::Equipmnt::Weapons::Axe` 같은 정규화된 이름으로 직접 참조하는 곳이 없어서 컴파일 에러는 안 났음
- 하지만 향후 정규화된 참조 시 컴파일 에러 발생하거나, 다른 개발자가 혼란을 겪을 수 있음
- GameplayTag 문자열("GameItems.Equipment.Weapons.Axe")은 .cpp의 `UE_DEFINE_GAMEPLAY_TAG`로 등록되므로 런타임 동작에는 영향 없음

### 테스트 방법
```
환경: 에디터 PIE
확인:
  1. 게임 시작 후 무기 아이템(도끼, 검 등) 습득
  2. 인벤토리에서 장비 카테고리 분류 정상 동작 확인
  3. 크래프팅 메뉴 장비 탭에 아이템 표시되는지 확인
  4. 빌드 성공 자체가 검증 완료 (네임스페이스 불일치 해소)
```

## 4. 서버 RPC Validate GridIndex/EntryIndex 상한 검증

### 파일
- `Plugins/Inventory/Source/Inventory/Public/InventoryManagement/Components/Inv_InventoryComponent.h`
- `Plugins/Inventory/Source/Inventory/Private/InventoryManagement/Components/Inv_InventoryComponent.cpp`

### 추가: 검증 상수 (헤더)
```cpp
namespace InvValidation
{
    constexpr int32 MaxGridIndex = 10000;
    constexpr int32 MaxEntryIndex = 10000;
}
```

### 수정 전
```cpp
// Server_UpdateItemGridPosition_Validate
return GridIndex >= INDEX_NONE && GridCategory <= 2;
// → GridIndex 상한 없음. 99999도 통과.

// Server_TakeItemFromContainer_Validate
return ContainerEntryIndex >= 0;
// → TargetGridIndex 검증 전혀 없음.

// Server_PutItemInContainer_Validate
return PlayerEntryIndex >= 0;
// → TargetGridIndex 검증 전혀 없음.
```

### 수정 후
```cpp
// Server_UpdateItemGridPosition_Validate
return GridIndex >= INDEX_NONE && GridIndex < InvValidation::MaxGridIndex && GridCategory <= 2;

// Server_TakeItemFromContainer_Validate
return ContainerEntryIndex >= 0 && ContainerEntryIndex < InvValidation::MaxEntryIndex
    && (TargetGridIndex == INDEX_NONE || (TargetGridIndex >= 0 && TargetGridIndex < InvValidation::MaxGridIndex));

// Server_PutItemInContainer_Validate
return PlayerEntryIndex >= 0 && PlayerEntryIndex < InvValidation::MaxEntryIndex
    && (TargetGridIndex == INDEX_NONE || (TargetGridIndex >= 0 && TargetGridIndex < InvValidation::MaxGridIndex));
```

### 문제였던 이유
- Validate는 네트워크 수신 시 값 타입만 검증하는 1차 방어선
- 상한 없이 `>= 0`만 체크하면 비정상 패킷(GridIndex=99999)이 통과
- 서버 _Implementation에서 `IsValidIndex()` 체크가 있지만, Validate에서 걸러주는 것이 UE 모범 사례
- Lobby에서는 이미 `LobbyValidation::MaxInventoryIndex(10000)` 패턴 사용 중 → 동일 패턴 적용
- 정상 게임플레이 (그리드 최대 ~200슬롯)에서는 절대 걸리지 않음

### 테스트 방법
```
환경: 데디케이티드 서버 + 클라이언트 3인
정상 동작 확인:
  1. 인벤토리 아이템 드래그&드롭 배치 → 정상
  2. 컨테이너(상자) 열기 → 아이템 가져오기/넣기 → 정상
  3. 아이템 Split(스택 분리) → 정상
  4. 서버 로그에서 Validate 실패(킥) 발생하지 않아야 함

비정상 패킷 차단 확인 (선택):
  - 서버 콘솔에서 비정상 GridIndex 로그가 없는지 확인
  - MaxGridIndex(10000) 초과 시 Validate가 false 반환 → RPC 거부
```

```
  3. 게임플레이 자체에는 영향 없음 (UE_LOG 출력만 비활성화)

다시 켜는 법:
  Helluna.h에서 필요한 플래그만 1로 변경 → 재빌드
  예: #define HELLUNA_DEBUG_WEAPON_BRIDGE 1
```
